# Project Architecture and Patterns Summary

* **Core Project Identity:** The project is a massive C++ and QML simulation framework containing 40+ modules (e.g., OmniKernel, OmniQuantum, OmniBio). It was formerly named "bobui" and is currently undergoing a complete rename to "bobqt". It also goes by the name BobQ or OmniUI in various contexts.
* **Architecture:** The framework heavily relies on Qt's Meta-Object System (`QObject`, `Q_PROPERTY`, `Q_INVOKABLE`, `QML_ELEMENT`, `QML_SINGLETON`) to establish a bridge between the C++ backend logic and the QML frontend user interfaces.
* **Framework Assimilation & Expansion:** BobQ is an ambitious project actively assimilating various other toolkits including Qt, GTK, JUCE, Ultimate++, JavaFX, WxWidgets, WinUI, and DearImgui.
* **Language Parity & Go Ports:** There is a strict mandate that every C++ UI feature must be ported to Go, maintaining 100% feature parity between the C++ and Go backends. A current high priority is completing the `OmniAudioGraph` DSP Go port.
* **Submodules & Dependencies:** Third-party libraries like JUCE and Ultimate++ are located in the `submodules/` directory. Building is handled via CMake (e.g., `mkdir build && cd build && cmake .. && make`). There is a known warning to be extremely cautious when updating `CMakeLists.txt` or `configure.cmake` due to parsing bugs related to orphaned `qt_internal` macros.
* **Development Workflows:**
  - Feature development is done module-by-module, updating both the QML UI and C++ backend concurrently.
  - UI representation must be extremely robust, requiring explicit interactive forms, clear labels, distinct descriptions, and detailed tooltips for all backend features.
  - Code must be commented in-depth explaining the 'why', structural side effects, and alternate methods attempted.
* **Documentation & Governance Standards:** The project enforces rigorous documentation standards. Key files include:
  - `VISION.md`: Ultimate goals and foundational concepts.
  - `MEMORY.md`: Ongoing internal architectural observations.
  - `DEPLOY.md`: Deployment and environment instructions.
  - `ROADMAP.md`: Long-term structural plans and milestones.
  - `TODO.md`: Short-term features and bug fixes.
  - `IDEAS.md`: Aggressive ideas for pivots, refactoring, or feature expansions.
  - `CHANGELOG.md` & `VERSION.md`: A single, global version string/date/build number is maintained in `VERSION.md` and referenced during commits.
  - `HANDOFF.md`: Detailed session history to enable frictionless model handoffs.
* **Git & Autonomy Mandates:** Autonomous execution is a core principle. The system dictates a strict repository sync protocol (upstream sync, branch merging, catch-up sync, submodule cleanup) before beginning work, and regular committing/pushing after every major feature without pausing for user confirmation unless explicitly requested.
